**Step 1: Verify SVAR drag-and-drop is enabled.**

SVAR React Gantt should support drag-to-link by default. If it does NOT work out of the box:
- Check if there's a config prop like `links: true` or `dragLinks: true` that needs to be set
- Consult SVAR docs at https://docs.svar.dev/react/gantt/guides/loading_data/ for link creation requirements
- The link drag gesture typically starts from a small circle/handle at the end of a task bar

**Step 2: Verify the `add-link` event fires correctly.**

In the `init` callback of GanttView.tsx, ensure:
```typescript
api.intercept('add-link', (ev) => {
  // ev.link should have { source, target, type }
  // source and target should be our UUID strings (task IDs)
  onAddLink?.(ev.link.source, ev.link.target);
  return false; // Prevent SVAR from managing the link internally
});
```

If the event structure differs from expected (e.g., `ev.link` is nested differently, or source/target are numeric), adjust the mapping. Log `ev` to console during development to inspect the shape.

**Step 3: Verify the full persistence flow.**

When a user drags from task A to task B:
1. SVAR fires `add-link` event with source=A.id, target=B.id
2. GanttView calls `onAddLink(A.id, B.id)`
3. page.tsx `handleAddLink` calls `addDependency(client, { projectId, upstreamTaskId: sourceId, downstreamTaskId: targetId })`
4. Supabase inserts the dependency row
5. `refetch()` reloads all data
6. `useSchedule` recomputes dates
7. `toSvarLinks` includes the new dependency
8. SVAR re-renders with the new arrow

**Step 4: Cycle detection.**

The `addDependency` call in `handleAddLink` already goes through our existing query layer. If the Supabase insert succeeds but creates a cycle, the `useSchedule` hook's `computeSchedule` will throw `CyclicDependencyError`. This is already caught and displayed as an error banner.

However, for a better UX, add pre-flight cycle detection in `handleAddLink` BEFORE calling `addDependency`:

```typescript
import { detectCycle } from '@/lib/scheduling/dependency-graph';

const handleAddLink = useCallback(async (sourceId: string, targetId: string) => {
  // Pre-flight cycle check
  const proposedDeps = [...dependencies, {
    id: 'temp',
    projectId: DEFAULT_PROJECT_ID,
    upstreamTaskId: sourceId,
    downstreamTaskId: targetId,
    dependencyType: 'finish-to-start' as const,
  }];
  const cycle = detectCycle(tasks.map(t => t.id), proposedDeps);
  if (cycle.length > 0) {
    toast.error(`Circular dependency: ${cycle.join(' → ')}`);
    return;
  }

  try {
    await addDependency(client, {
      projectId: DEFAULT_PROJECT_ID,
      upstreamTaskId: sourceId,
      downstreamTaskId: targetId,
    });
    await refetch();
    toast.success('Dependency created');
  } catch (err) {
    toast.error(err instanceof Error ? err.message : 'Failed to create dependency');
  }
}, [client, tasks, dependencies, refetch]);
```

Check the `detectCycle` function signature in `src/lib/scheduling/dependency-graph.ts`. It may expect different argument shapes -- read the file and adapt. The key is: pass current task IDs + proposed new dependency to the cycle detector.
